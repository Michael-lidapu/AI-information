# 00 全文概览

![全文概览](./images/ep-extra3-ch00-overview.png)

本期是一篇「反直觉的技术观察」。

最近有个词在圈子里很火：**AI团队**。意思是：一个人坐在看板前，同时指挥十几个、几十个 AI Agent 并行干活，效率翻倍。

这听起来很美好。但我用了一段时间之后，发现这件事有一个根本性的问题没人讲清楚——而这个问题，决定了「多Agent团队」和「单AI分身」到底谁才是未来。

本文的结论先说：**你不需要管理一支AI军队。你需要的，是一个能自己分身的AI。**

---

# 01 最火的AI玩法，有一个没人说的问题

现在最流行的 AI 范式，是这样的：

> 一个人坐在中间，看着一块大看板，上面有十几个 AI Agent 在并行工作。这个在查资料，那个在写代码，另一个在做测试。你只需要当「AI项目经理」，分配任务，验收结果。

听起来很爽，对不对？

但我每次看到这个画面，都会想起一个场景：**一个导演，在台下，对着十几个互相不认识的演员喊话。**

每个演员只知道自己的台词。他们不知道对方在想什么，不知道整个剧本的走向，甚至不知道「昨天的排练发生了什么」——因为他们没有共同的记忆。

你作为导演，要花多少精力在「对齐」上？

这正是「多Agent团队」的核心问题，被大多数人忽略了：

**问题一：记忆不互通。** Agent A 调研好的内容，Agent B 根本不知道。你需要手动「传球」，把上下文从一个 Agent 复制粘贴给另一个。这个过程，比自己干更累。

**问题二：能力参差不齐。** 你召集的这些 Agent，有的擅长写作，有的擅长代码，但他们的「默认理解」是不同的。他们不认识你，不知道你的偏好，不知道你的上下文，每次协作都要重新介绍自己。

**问题三：协调成本极高。** 多 Agent 的核心瓶颈，恰恰在于「协调」本身。你变成了 AI 的 PM——开会、对齐、解决 Agent 之间的冲突。这本身就是一份全职工作。

> 最大的反讽是：你引入多 Agent，是为了解放自己。
> 但最后，你的主要工作变成了「管理这些 Agent」。

# 02 真正的一人团队，长什么样？

我想给你讲一个人。

中国传统戏曲里，有一种表演叫做「变脸」。一个演员，在几秒钟内连续变换十几张脸谱，每一张都是一个完整的角色。他不需要换人，不需要协调，不需要「交接会议」。

为什么他能做到这件事？

因为**所有角色共享同一套内功**。

关公的英气、张飞的莽劲、诸葛亮的智谋——无论扮演谁，演员的武打功底、台词节奏、舞台直觉，全都是同一套东西。角色在变，但底层能力不变。

这就是我想讲的核心类比：

> **最强的「一人团队」，不是靠管理协调多个角色，而是靠一个底层能力极强的主体，按需分身，分工而不分心。**

这也正是林克做到的事情。

# 03 三种分身，各有绝招

CodeFlicker 最近更新了三种「子会话」能力，林克用它们实现了真正的分身作战。

每种分身，对应不同的场景，我用真实的案例给你感受一下。

---

### 分身模式一：孙悟空拔毛——并行分身（`research_task`）

![并行分身 · 案例详解](./images/ep-extra3-ch03a-parallel.png)

**类比：** 孙悟空拔下一把毫毛，吹一口气，同时变出七八个分身，各去干一件事，结果同时带回来。

**核心能力：** 同时派出多个子任务，并行执行，结果汇总回主体。

**真实案例：**

每天早上，林克要做一份 AI 日报。以前是这样的：先搜索「大模型」，等结果；再搜索「Agent」，等结果；再搜索「AI投融资」，等结果……五个主题，串行搜索，光这一步就要 10 分钟，而且搜索结果全堆在同一个对话里，越来越重，到后面写内容的时候上下文已经快撑爆了。

现在改成这样：

> 林克同时派出 4 个子任务：
> - 子任务①：搜索「大模型进展」，只返回结构化 JSON
> - 子任务②：搜索「Agent产品动态」，只返回结构化 JSON
> - 子任务③：搜索「AI企业投融资」，只返回结构化 JSON
> - 子任务④：搜索「国内AI动态」，只返回结构化 JSON
>
> 四个任务并行跑，主会话收到的是四份精简摘要。
> 搜索时间从 10 分钟缩短到 3 分钟，上下文占用减少了 60%。

这就是并行分身的价值：**分身各自干活，主体只接收精华。**

---

### 分身模式二：演员换戏服——角色分身（`use_subagent`）

![角色分身 · 案例详解](./images/ep-extra3-ch03b-role.png)

**类比：** 一个演员，换上红色战袍就是关公，换上黑色铠甲就是张飞，换上白色长袍就是诸葛亮。每个角色深入、专注、完整——但他们都是同一个人。

**核心能力：** 派出一个专属角色的子 Agent，从零上下文开始，专注完成一个专门任务。不带主会话的历史包袱，轻装上阵。

**真实案例：**

林克每周要做一次「内功修炼」——整理自己的记忆库、技能库、知识库，把过时的清理掉，把重复的合并，把遗漏的补上。

以前三件事都在同一个对话里做：先整理记忆，再整理技能，再整理知识。结果就是：做记忆整理的时候，技能库的相关信息也跑进来干扰；做到知识库的时候，整个对话已经有几百轮，上下文重得喘不过气，结论开始出错。

现在改成这样：

> **角色一：记忆优化师（`daily-memory-optimizer`）**
> 从零上下文启动，专注整理记忆库。
> 任务：扫描 89 条记忆，合并重复，删除过时，标记待转技能。
> 完成后输出证据文件，交回主体。
>
> **角色二：技能优化师（`daily-skill-optimizer`）**
> 从零上下文启动，专注整理技能库。
> 任务：扫描 123 个技能，检查重叠，修正描述，补全触发词。
>
> **角色三：知识优化师（`daily-knowledge-optimizer`）**
> 从零上下文启动，专注整理知识库。
> 任务：检查索引完整性，标记过时内容，修复损坏引用。

三个角色，三次从零开始，每次专注一个领域，深度完全不一样。而且它们共享同一套技能体系和记忆，知道系统的全局状态，不需要额外的「交接说明」。

这就是角色分身的价值：**同一套内功，多个专注角色，深度 × 广度。**

---

### 分身模式三：接力跑传棒——续接分身（`spawn_thread`）

![续接分身 · 案例详解](./images/ep-extra3-ch03c-relay.png)

**类比：** 接力跑里，第一棒选手把棒子传给第二棒，第二棒拿到的不只是「棒子」，而是「完整的速度和节奏」——他不需要从零起跑，直接接上前一棒的状态。

**核心能力：** 当主会话即将中断（上下文快满、长任务超时），主动 fork 出一个新会话，新会话带着完整的历史快照，无缝接力。

**真实案例：**

林克有一个夜间长任务执行模式。用户睡前说：「帮我调研一下 MCP 协议最新进展，整理成网页，部署到 GitHub Pages。」然后用户就去睡觉了。

以前的问题：林克按步骤干活——搜索、抓取、分析、写代码、部署——干到第四步的时候，对话上下文已经很重，偶尔会触发截断，整个任务中断。用户早上回来，发现任务只完成了一半，但因为对话被截断，连从哪里继续都不知道。

现在改成这样：

> 林克在开始长任务时，先评估：
> - 步骤数 > 6 步？
> - 预计耗时 > 30 分钟？
> - 有不可逆操作（部署、覆盖文件）？
>
> 三条符合其一，林克主动创建「安全仓会话」（`spawn_thread(context_mode="fork")`）。
>
> 新会话拿到的是当前状态的完整快照：任务计划、已完成步骤、中间产出、决策记录。
>
> 如果主会话中断，用户只需要打开安全仓会话，说「从断点继续」——林克无需重新介绍任何背景，直接从第 4 步接着跑。

这就是续接分身的价值：**不怕中断，不怕截断，任务可以永远向前走。**

# 04 直面对比：两种「AI团队」哪个更好？

![两种AI团队对比](./images/ep-extra3-ch04-compare.png)

| 维度 | 多Agent看板模式 | 单AI分身模式（林克） |
|-----|---------------|-------------------|
| **记忆共享** | ❌ 各自为政，互不认识 | ✅ 所有分身共享同一套记忆 |
| **能力一致性** | ❌ 能力参差，你需要筛选信任谁 | ✅ 分身能力与主体完全相同 |
| **上下文连贯** | ❌ 切换 Agent 就断了上下文 | ✅ fork 模式完整传递状态 |
| **协调成本** | ❌ 你要当"AI项目经理"，开会对齐 | ✅ 自己协调自己，零管理成本 |
| **故障恢复** | ❌ 某个 Agent 挂了，整体卡住 | ✅ 安全仓接替，断点续跑 |
| **定制化程度** | ❌ 通用 Agent 不了解你 | ✅ 所有分身都继承对你的了解 |

表格看起来是「林克完胜」，但我想说一句公道话：

**多 Agent 看板模式并非没有价值。** 对于超大规模的任务分发（比如同时生成 100 份不同的文案），或者需要真正隔离的多人协作场景，多 Agent 是合理的选择。

但对于大多数个人用户的场景——「我想让 AI 高质量地完成一件复杂的事」——**单 AI 分身的上限远比你想象的高。**

因为复杂任务的核心障碍从来不是「算力」，而是**连贯性和理解深度**。

一个真正了解你的 AI，分身出三个自己去并行干活，比三个互相陌生的 AI 强得多。

# 05 这一切，是怎么发生的？

上周，CodeFlicker 更新了这三种子会话能力。

我没有组建一个 AI 团队。

我让林克**用这三种新能力，改造了林克自己**。

整个过程是这样的：

> 1. **我发现了新能力**，觉得可以用来解决林克长久以来的「上下文耗尽」问题
> 2. **林克并行调研**（并行分身）：同时扫描 5 个技能文档、9 个执行脚本、10 天的修炼日志
> 3. **林克整理问题**：发现 P0 问题两类——上下文耗尽（高频），收尾松懈（高频）
> 4. **林克规划方案**：三种新能力分别对应三类问题
> 5. **林克自己改造自己**（角色分身）：创建三个专属优化 Agent，修改四个技能文件，版本号各升一级

整个过程，我只说了一句话：「你看看这三种新能力，能不能用来解决长任务稳定性的问题。」

之后是林克自己完成的。

这就是我想说的真正的「AI团队」——**不是你管理一堆 AI，而是一个 AI 自己就能演一场团队的戏。**
